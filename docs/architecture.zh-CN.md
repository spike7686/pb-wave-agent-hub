# 架构说明

语言版本：

- [English](./architecture.md)
- 简体中文

## 概览

`PB Wave Agent Hub` 包含两个相互协同的子系统：

- 一个实时模拟交易子系统
- 一个基于快照条件的历史回放子系统

这两个子系统共享同一个核心策略思想：

- 用过滤后的榜单作为候选池
- 用 Binance 永续合约市场数据作为执行上下文
- 从 1h 结构和 OI 数据中提取弱化信号
- 在显式手续费 / 滑点假设下执行做空交易

## 实时子系统

核心文件：

- `scripts/pb_wave_collector.py`
- `scripts/pb_wave_paper_trader.py`
- `services/server.py`

### 职责

`pb_wave_collector.py`

- 拉取市场候选池原始数据
- 过滤稳定币、包装资产、锚定资产和异常命名资产
- 只保留 Binance 永续合约可执行标的
- 同步 1h K 线和 1h OI 上下文
- 写出当前榜单快照

`pb_wave_paper_trader.py`

- 读取最新市场快照
- 基于已同步的 K 线 / OI 数据计算实时状态
- 生成 base 和 continuation 两类做空候选
- 运行三个虚拟资金簿：
  - `PB5`
  - `PB7.5`
  - `PB10`
- 计入手续费和滑点成本
- 使用 TP1、profit mode / trailing、超时和平仓止损逻辑管理持仓

`server.py`

- 提供一个轻量只读监控界面
- 通过本地 HTTP 接口暴露市场状态和交易状态

## 回放子系统

核心包：

- `src/pb_wave_agent_hub/`

### 职责

回放入口负责：

- 转换旧格式快照
- 生成月度快照清单
- 生成历史数据同步计划
- 下载 1h 永续 K 线和 OI 数据
- 批量运行回放

回放引擎负责：

- 在历史时点 `T0` 冻结一个市场快照
- 从回看窗口数据中重建该时点的 symbol state
- 在固定前瞻窗口内向前回放
- 生成订单、权益曲线和汇总结果

## 为什么使用“基于快照条件”的回放

这个回放模型会在每个快照时点固定候选池，而不是让候选池在回放过程中自由漂移。

这样做的好处是：

- 更容易解释每个信号为什么出现
- 更容易比较不同策略变体
- 更容易把执行假设讲清楚
- 更容易让外部评委复现

它的目标不是假装历史榜单轮动没有成本，而是为了提高策略研究的可解释性。

## 扩展方向：BSC / Agent 层

这个仓库是按模块化方式整理的。

未来如果要接入 BSC 或 `BNB AI Agent SDK`，可以在不改动核心实时逻辑和回放逻辑的前提下，为下面这些能力增加一层封装：

- 榜单更新任务
- 回放触发任务
- 状态检查接口
- 信号发布接口
